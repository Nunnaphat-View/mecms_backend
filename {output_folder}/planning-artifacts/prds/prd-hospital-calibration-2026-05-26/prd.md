# ระบบสอบเทียบและบำรุงรักษาเครื่องมือแพทย์ระดับโรงพยาบาล

## 1. วิสัยทัศน์และวัตถุประสงค์ (Vision & Objectives)

### 1.1 วิสัยทัศน์หลัก (Core Vision)
พัฒนาระบบการทำงานแบบดิจิทัลทดแทนการใช้กระดาษ (Paperless Workflow) สำหรับขั้นตอนการทำบำรุงรักษาเชิงป้องกัน (Preventive Maintenance - PM) และการสอบเทียบเครื่องมือแพทย์ (Calibration) เพื่อลดความผิดพลาดในการกรอกข้อมูลจากเจ้าหน้าที่ (Human Error) เพิ่มความเที่ยงตรง ความน่าเชื่อถือ และความปลอดภัยสูงสุดในการรักษาผู้ป่วย

### 1.2 ตัวชี้วัดความสำเร็จ (Success Metrics)
* **การออกใบรายงานทันที (Instant Report Generation):** สามารถสร้างและออกใบรายงานผลสอบเทียบได้ทันทีเมื่อได้รับการอนุมัติผลลัพธ์
* **ประวัติเครื่องมือย้อนกลับได้ 100% (100% Traceability):** ประวัติและข้อมูลผลการสอบเทียบทั้งหมดถูกเก็บอย่างเป็นระบบ สามารถค้นหาผ่านรหัสหรือการสแกน QR Code ได้รวดเร็ว
* **ลดเวลาการหยุดทำงานของเครื่องมือ (Downtime Reduction):** การระบุสถานะของเครื่องมือแพทย์ที่ชัดเจนและรวดเร็วช่วยลดความสับสนในการใช้งานเครื่องที่ชำรุด

### 1.3 ข้อจำกัดและความปลอดภัยหลัก (Key Constraints & Security)
* **ห้ามแก้ไขประวัติย้อนหลัง (Immutability of Logs):** เมื่อผลการสอบเทียบหรือประวัติการซ่อมได้รับการอนุมัติและสิ้นสุดกระบวนการแล้ว ข้อมูลเหล่านั้นจะไม่สามารถแก้ไขหรือลบออกย้อนหลังได้ เพื่อรักษาความสมบูรณ์ในการสืบทวนสอบมาตรฐานโรงพยาบาล

## 2. คุณสมบัติและฟังก์ชันหลักของระบบ (Core Features)

### 2.1 ระบบระบุตัวตนและตรวจสอบข้อมูลเครื่องมือแพทย์ (QR Code & Inventory Lookup)
* **การสแกน QR Code (Scan to Action):** รองรับการสแกน QR Code เพื่อตรวจสอบรายละเอียดทางเทคนิคของเครื่องมือแพทย์ สถานะปัจจุบัน (เช่น พร้อมใช้งาน, รอซ่อม, หยุดใช้งาน, จำหน่าย) และประวัติกิจกรรมล่าสุด
* **การสืบค้นประวัติย้อนหลัง (Historical Audit Trail):** การเรียกดูประวัติของเครื่องมือที่ผ่านการทำ PM, งานซ่อมแซม และการสอบเทียบ (Calibration) ทั้งหมดแบบดิจิทัล

### 2.2 ระบบบำรุงรักษาเชิงป้องกัน (Preventive Maintenance - PM Module)
* **การบันทึก PM เช็คลิสต์ (Digital PM Checklist):** ช่างสามารถเลือกเครื่องมือแพทย์และบันทึกผลการตรวจสอบตามหัวข้อของเช็คลิสต์มาตรฐาน
* **การจัดการสถานะเมื่อ PM ไม่ผ่าน (PM Failure Handling):** หากช่างระบุผลการตรวจสอบ PM มีข้อที่ไม่ผ่านเกณฑ์ขั้นวิกฤต ระบบจะบังคับอัปเดตสถานะของเครื่องมือแพทย์เป็น **"รอซ่อม"** และปิดกระบวนการงานรอบปัจจุบันโดยอัตโนมัติ

### 2.3 ระบบสอบเทียบเครื่องมือวัด (Calibration Module)
* **การคำนวณและวัดผลแบบไดนามิก (Quantitative Test Entries):** บันทึกค่าข้อมูลการวัดสอบเทียบ และเทียบกับค่ามาตรฐานความถูกต้อง (Acceptance Criteria / Calibration Settings)
* **การจัดการสถานะเมื่อสอบเทียบไม่ผ่าน (Calibration Failure Handling):** หากระบบคำนวณหรือช่างบันทึกผลว่าการสอบเทียบไม่ผ่านเกณฑ์มาตรฐาน ระบบจะปรับสถานะเครื่องมือเป็น **"หยุดใช้งานเครื่อง"** ทันที เพื่อป้องกันบุคลากรทางการแพทย์สับสนนำไปใช้งานกับคนไข้

### 2.4 ระบบอนุมัติผลและทวนสอบงานสองขั้นตอน (Multi-level Approval Workflow)
* **การส่งอนุมัติให้หัวหน้าแผนก (Supervisor Verification Gate):** เมื่องาน PM/Calibration เสร็จสิ้น ข้อมูลทั้งหมดจะถูกส่งให้หัวหน้าแผนกตรวจสอบแบบออนไลน์
* **การจัดการคำขอที่ไม่อนุมัติ (Rejection/Rework Flow):** หากหัวหน้าแผนกตรวจพบข้อบกพร่อง สามารถไม่อนุมัติและส่งกลับไปที่ช่างเพื่อทำ PM หรือสอบเทียบใหม่ (สถานะระบบจะยังคงเดิมจนกว่าจะตรวจประเมินใหม่)
* **การอนุมัติเคสผ่าน (Calibration Pass Approval):** หากได้รับการอนุมัติในเคสที่ผ่านการสอบเทียบ ระบบจะทำการเปิดสิทธิ์ให้ออก QR Code ใหม่สำหรับติดประจำตัวเครื่อง และอัปเดตสถานะเป็น **"พร้อมใช้งาน"**

### 2.5 ระบบพิจารณาจัดการเครื่องชำรุดระดับผู้บริหาร (Executive Disposition Module)
* **การพิจารณาเครื่องที่สอบเทียบไม่ผ่าน (Director Decision Gate):** กรณีที่ผลการสอบเทียบประเมินว่าไม่ผ่านและหัวหน้าแผนกอนุมัติผลลัพธ์นั้น งานจะส่งต่อให้ ผู้อำนวยการ (ผอ.) เป็นผู้พิจารณาตัดสินใจขั้นสุดท้าย
* **การจำหน่ายหรือซ่อม (Dispose vs Repair Decision):**
  * หาก ผอ. ตัดสินใจให้ **"ซ่อม"** ➡️ ระบบจะเปลี่ยนสถานะเครื่องเป็น **"รอซ่อม"** เพื่อแจ้งส่งช่างซ่อมบำรุง
  * หาก ผอ. ตัดสินใจให้ **"ขาย/ทำลาย"** ➡️ ระบบจะเปลี่ยนสถานะเครื่องเป็น **"จำหน่าย"** (Decommissioned) เพื่อออกจากบัญชีใช้งานโรงพยาบาลถาวร

### 2.6 ระบบออกใบรายงานและบันทึกประวัติความปลอดภัย (Reporting & Immutable Log System)
* **การสร้างเอกสาร PDF (Instant Certificate Generation):** การแปลงผลการสอบเทียบเป็นเอกสารประเมินผลสอบเทียบ (Calibration Certificate) ในรูปแบบ PDF ทันทีหลังจากการอนุมัติเสร็จสิ้น
* **ประวัติที่ปลอดภัยและไม่สามารถแก้ไขได้ (Immutable Audit Logs):** ข้อมูลผลการสอบเทียบและเช็คลิสต์ที่ได้รับการอนุมัติแล้วจะถูกบันทึกไว้อย่างถาวร ไม่สามารถเข้าถึงเพื่อแก้ไขหรือลบจากหน้าบ้านหรือโดยผู้ใช้งานคนใด เพื่อเป็นประวัติสืบทวนสอบมาตรฐาน JCI หรือการตรวจสอบภายนอก

## 3. บทบาทผู้ใช้งานและเส้นทางการทำงาน (User Roles & Journeys)

เพื่อสะท้อนการทำงานจริงในระบบ เรากำหนดตัวละครสมมติ (Protagonists) ของระบบผ่าน 3 บทบาทหลักดังนี้:

### 3.1 คุณสมชาย — ช่างสอบเทียบ (Calibration Technician)
* **บริบท:** คุณสมชายต้องดูแลตรวจสอบเครื่องมือแพทย์หลายสิบชิ้นต่อวันในหอผู้ป่วยต่าง ๆ เขาต้องการเครื่องมือที่ช่วยให้สแกนบันทึกผลได้ง่ายและลดการคีย์กระดาษ
* **เส้นทางการทำงาน (User Journey - UJ-1):**
  1. คุณสมชายถือแท็บเล็ตเดินไปยังหอผู้ป่วยเด็กเพื่อเริ่มงาน PM รอบเช้า
  2. เขาใช้กล้องสแกน **QR Code** บนเครื่องวัดความดันเด็ก เพื่อระบุข้อมูลตัวเครื่องในระบบทันที
  3. คุณสมชายทำ PM ตามเช็คลิสต์บนหน้าจอ หากพบสายไฟชำรุดร้ายแรง เขาบันทึกผลว่า **"ไม่ผ่าน"** ➡️ ระบบจะเปลี่ยนสถานะเครื่องเป็น **"รอซ่อม"** ทันที และเขาสามารถจบงานเพื่อส่งซ่อมได้เลย
  4. หาก PM ผ่านทั้งหมด เขาทำขั้นตอน **Calibration** (สอบเทียบ) ต่อ โดยนำเครื่องมือมาตรฐานมาร่วมวัดผล และกรอกค่าตัวเลขลงในระบบ
  5. หากผลการสอบเทียบคำนวณแล้ว **"ผ่าน"** หรือ **"ไม่ผ่าน"** เขาจะกดปุ่ม **"ส่งข้อมูลให้หัวหน้าแผนกตรวจสอบ"** ➡️ สถานะของเครื่อง (หากไม่ผ่าน) จะเปลี่ยนเป็น **"หยุดใช้งานเครื่อง"** เพื่อรอการรีวิว

### 3.2 คุณศิริพงษ์ — หัวหน้าแผนกสอบเทียบ (Department Head)
* **บริบท:** คุณศิริพงษ์มีหน้าที่ตรวจสอบความถูกต้องของการทำ PM และ Calibration ของช่างทุกคนให้ตรงตามมาตรฐาน ISO 13485 ก่อนออกใบรายงานผลสอบเทียบอย่างเป็นทางการ
* **เส้นทางการทำงาน (User Journey - UJ-2):**
  1. คุณศิริพงษ์เปิดระบบบราวเซอร์ที่หน้าจอของเขา และเห็นแจ้งเตือนรายการรออนุมัติของเครื่องวัดความดันเด็กที่คุณสมชายเพิ่งส่งมา
  2. เขาตรวจสอบบันทึกผล PM และตารางผลการวัดสอบเทียบอย่างละเอียด
  3. **กรณีไม่อนุมัติ (เช่น ค่าเบี่ยงเบนยังผิดปกติ):** เขาระบุเหตุผลและกด **"ไม่อนุมัติ"** ➡️ ระบบจะส่งงานกลับไปยังกล่องงานของคุณสมชายเพื่อทำ PM/Calibration ใหม่
  4. **กรณีอนุมัติ (ผลสอบเทียบผ่าน):** เขากด **"อนุมัติ"** ➡️ ระบบจะเปลี่ยนสถานะเครื่องมือเป็น **"พร้อมใช้งาน"**, สร้าง QR Code สำหรับการสอบเทียบรอบใหม่ และออกใบรายงานผลสอบเทียบ (PDF Calibration Certificate) เข้าระบบคลังเอกสารอัตโนมัติเพื่อให้พิมพ์หรือดาวน์โหลดได้ทันที
  5. **กรณีอนุมัติ (ผลสอบเทียบไม่ผ่าน):** เขากด **"อนุมัติผลไม่ผ่าน"** ➡️ ระบบจะส่งผลการตรวจสอบนี้ต่อไปยังระดับ ผอ. เพื่อพิจารณาการซ่อม/จำหน่าย

### 3.3 ผอ. วิชัย — ผู้อำนวยการฝ่ายการแพทย์ (Medical Director)
* **บริบท:** ผอ. วิชัย เป็นผู้รับผิดชอบการตัดสินใจในการบริหารสินทรัพย์เครื่องมือแพทย์ของโรงพยาบาล โดยเน้นความคุ้มค่าและไม่ให้เครื่องมือที่เสื่อมสภาพหลุดไปใช้งานกับคนไข้
* **เส้นทางการทำงาน (User Journey - UJ-3):**
  1. ผอ. วิชัย ได้รับรายงานว่ามีเครื่องวัดความดันเด็กที่ไม่ผ่านการสอบเทียบ และผ่านการยืนยันผลโดยคุณศิริพงษ์แล้ว
  2. เขาเปิดหน้าพิจารณาของเครื่องมือแพทย์ชิ้นนั้น เพื่อดูประวัติย้อนหลังว่าเคยซ่อมมาแล้วกี่ครั้ง คุ้มค่าแก่การซ่อมต่อหรือไม่
  3. **กรณีตัดสินใจซ่อม:** เขากดปุ่ม **"อนุมัติส่งซ่อม"** ➡️ ระบบจะเปลี่ยนสถานะเครื่องเป็น **"รอซ่อม"** เพื่อเริ่มงานในฝั่งช่างซ่อมบำรุง
  4. **กรณีตัดสินใจจำหน่าย:** เขากดปุ่ม **"อนุมัติจำหน่ายเครื่อง"** ➡️ ระบบจะอัปเดตสถานะของเครื่องเป็น **"จำหน่าย"** (Decommissioned/Scrapped) เป็นการตัดสถานะเครื่องมือออกจากระบบถาวร ไม่สามารถใช้งานในระบบตรวจสอบหรือสืบค้นเพื่อสอบเทียบได้อีกต่อไป

## 4. ข้อกำหนดการทำงานของระบบ (Functional Requirements)

### 4.1 ระบบระบุตัวตนและข้อมูลเครื่องมือแพทย์ (QR & Inventory Lookup)
* **FR-1.1 (QR Scanning):** ระบบต้องรองรับการใช้กล้องของอุปกรณ์เพื่อสแกน QR Code แล้วนำทางไปยังหน้าบันทึกผลหรือหน้าข้อมูลเครื่องมือแพทย์ชิ้นนั้นโดยตรง
* **FR-1.2 (Manual Search):** ระบบต้องมีฟังก์ชันค้นหาเครื่องมือวัดด้วยการกรอกรหัสเครื่องมือ (Equipment Code) หรือหมายเลขเครื่อง (Serial Number) โดยมีระบบ Auto-complete ช่วยค้นหา
* **FR-1.3 (Status Visualization):** ระบบต้องแสดงสถานะปัจจุบันของเครื่องมือแพทย์อย่างเด่นชัดด้วยสีSemantic (เช่น พร้อมใช้งาน = สีเขียว, รอซ่อม = สีฟ้า/เหลือง, หยุดใช้งานเครื่อง = สีแดง, จำหน่าย = สีเทา)

### 4.2 ระบบเช็คลิสต์บำรุงรักษาเชิงป้องกัน (PM Workflow)
* **FR-2.1 (Dynamic PM Template):** ระบบต้องโหลดชุดเช็คลิสต์การบำรุงรักษาเชิงป้องกันที่ตรงตามประเภทเครื่องมือแพทย์ชิ้นนั้นขึ้นมาแสดงโดยอัตโนมัติ
* **FR-2.2 (PM Fail Action):** หากข้อกำหนดของ PM มีข้อใดข้อหนึ่งที่เป็นหัวข้อวิกฤต (Critical Checkpoint) ไม่ผ่านเกณฑ์ ระบบต้องปรับสถานะเครื่องเป็น **"รอซ่อม"** ทันที และบังคับจบงานเพื่อส่งข้อมูลเข้าสู่ระบบแจ้งซ่อมโดยไม่ต้องทำขั้นตอน Calibration

### 4.3 ระบบบันทึกและคำนวณผลการสอบเทียบ (Calibration Workflow)
* **FR-3.1 (Standard Tool Association):** ระบบต้องให้ช่างเลือกเครื่องมือมาตรฐาน (Standard Tool) ที่ใช้ในการทดสอบวัดผล โดยต้องตรวจสอบว่าเครื่องมือมาตรฐานนั้นมีสถานะพร้อมใช้งานและไม่หมดอายุการสอบเทียบ (Expired Calibration Date)
* **FR-3.2 (Quantitative Parameters Input):** ระบบต้องแสดงตารางกรอกผลการวัดสอบเทียบตามที่กําหนดไว้ใน Calibration Setting ของเครื่องมือประเภทนั้น ๆ
* **FR-3.3 (Automated Calculation):** ระบบต้องคำนวณค่าคลาดเคลื่อน (Error / Uncertainty) อัตโนมัติทันทีหลังกรอกข้อมูลเสร็จสิ้น และประเมินเบื้องต้นว่าผ่านเกณฑ์ยอมรับ (Acceptance Criteria) หรือไม่
* **FR-3.4 (Calibration Fail Action):** หากผลการคำนวณสอบเทียบระบุว่าไม่ผ่านเกณฑ์การยอมรับ ระบบจะเปลี่ยนสถานะเครื่องมือเป็น **"หยุดใช้งานเครื่อง"** ทันทีหลังจากช่างส่งรายงานผลการวัด

### 4.4 ระบบกระบวนการอนุมัติสองขั้นตอน (Multi-level Approval Flow)
* **FR-4.1 (Supervisor Inbox):** ระบบต้องมีแดชบอร์ดเฉพาะสำหรับหัวหน้าแผนก เพื่อตรวจสอบรายการรอรีวิวทั้งหมด (Pending Approvals)
* **FR-4.2 (Approve & PDF Generation):** เมื่อหัวหน้าอนุมัติผลสอบเทียบที่ผ่านเกณฑ์ ระบบต้องอัปเดตสถานะเป็น **"พร้อมใช้งาน"**, สร้าง QR Code ข้อมูลรอบการสอบเทียบปัจจุบัน และออกรายงาน PDF Calibration Certificate เข้าระบบคลังประวัติโดยอัตโนมัติ
* **FR-4.3 (Rework/Rejection):** หัวหน้าสามารถเลือกส่งกลับ (Reject) ไปให้ช่างดำเนินการแก้ไข โดยกรอกบันทึกเหตุผลประกอบเพื่อให้ช่างเห็นจุดบกพร่อง
* **FR-4.4 (Escalate to Director):** กรณีที่หัวหน้าอนุมัติรายงานผลสอบเทียบที่เป็น **"ไม่ผ่าน"** (Calibration Failed) ระบบต้องส่งต่อเคสดังกล่าวไปยังขั้นตอนการพิจารณาของ ผอ. โดยอัตโนมัติ

### 4.5 ระบบพิจารณาจัดการสินทรัพย์ชำรุด (Director Disposition)
* **FR-5.1 (Director Decision Screen):** ผอ. ต้องมีหน้าจอเข้าพิจารณาเครื่องที่สอบเทียบไม่ผ่าน โดยระบบจะดึงประวัติการซ่อมบำรุงในอดีต (Maintenance History) และสถิติความเสียหายมาแสดงประกอบการตัดสินใจ
* **FR-5.2 (Disposition Action):** ผอ. สามารถเลือกดำเนินการได้ 2 รูปแบบ:
  * **ซ่อม (Send to Repair):** สถานะเครื่องจะปรับเป็น **"รอซ่อม"** และบันทึกข้อมูลเข้าระบบแจ้งซ่อม
  * **จำหน่าย (Decommission):** สถานะเครื่องจะปรับเป็น **"จำหน่าย"** ระบบจะปิดสิทธิ์การนำเครื่องมือนี้ไปดำเนินการสอบเทียบหรือบันทึกข้อมูลอื่น ๆ อีกต่อไป

### 4.6 ระบบส่งออกรายงานและข้อมูลประวัติถาวร (Reporting & Logs)
* **FR-6.1 (PDF Individual/Bulk Export):** ระบบต้องรองรับการส่งออกไฟล์เอกสาร PDF ทั้งแบบทีละใบ และแบบเลือกหลายใบพร้อมกันเพื่อดาวน์โหลดเป็นไฟล์ ZIP
* **FR-6.2 (Immutable Record Locking):** ข้อมูลผลการสอบเทียบและ PM ที่ได้รับการอนุมัติสิ้นสุดกระบวนการแล้ว จะถูกล็อกเป็นประวัติถาวร ห้ามไม่ให้ผู้ใช้หรือผู้ดูแลระบบทั่วไปทำการแก้ไขหรือลบย้อนหลังผ่านทางหน้าจอระบบ (Read-only)

## 5. ข้อกำหนดด้านประสิทธิภาพและความปลอดภัย (Non-Functional Requirements)

### 5.1 ด้านสิทธิ์และการเข้าถึงข้อมูล (Security & RBAC)
* **NFR-1.1 (Role-Based Access Control):** ระบบต้องควบคุมการเข้าถึงและการมองเห็นปุ่มการกระทำตามบทบาทผู้ใช้ (เช่น ช่างสอบเทียบเท่านั้นที่บันทึกผลได้, หัวหน้าแผนกเท่านั้นที่กดยอมรับ/ส่งต่อได้, ผอ. เท่านั้นที่กดสั่งจำหน่ายเครื่องมือได้)
* **NFR-1.2 (Secure Passwords & JWT):** รหัสผ่านของผู้ใช้ในระบบต้องถูกเข้ารหัสด้วยกลไกความปลอดภัยสูง (เช่น Bcrypt) และการสื่อสาร API ทั้งหมดต้องยืนยันตัวตนผ่าน JWT Token

### 5.2 ด้านความถูกต้องของข้อมูล (Data Integrity & Transaction Safety)
* **NFR-2.1 (Atomic Database Transactions):** การอัปเดตสถานะของเครื่องมือแพทย์พร้อมกับการบันทึกประวัติการบำรุงรักษาและการจัดสรรคิวงานอนุมัติต้องเสร็จสิ้นพร้อมกันทั้งหมดในระดับฐานข้อมูล (ACID Transaction) หากมีส่วนใดผิดพลาดระบบต้อง Rollback ข้อมูลกลับทั้งหมดเพื่อป้องกันข้อมูลขัดแย้งกัน

### 5.3 ด้านการออกแบบและประสบการณ์ผู้ใช้ (Usability & Design)
* **NFR-3.1 (Clean Industrial UI Layout):** ส่วนติดต่อผู้ใช้ต้องเป็นไปตามแนวทาง Clean Industrial (เช่น การใช้สี Semantic เท่าที่จำเป็น ไม่ตกแต่งสีสันเกินควร, ใช้ฟอนต์ Sarabun เป็นหลักเพื่อการอ่านที่ชัดเจน, แสดงผลในรูปแบบ Light Mode เท่านั้นเพื่อใช้งานได้ง่ายภายใต้แสงธรรมชาติของแผนก)
* **NFR-3.2 (Touch Optimization):** หน้าจอบนอุปกรณ์แท็บเล็ตและมือถือต้องมีขนาดปุ่มและพื้นที่กดสัมผัสไม่ต่ำกว่า 48x48 พิกเซล เพื่อให้ช่างทำงานขณะสวมถุงมือแพทย์ได้สะดวก

### 5.4 ด้านประสิทธิภาพการทำงาน (Performance)
* **NFR-4.1 (Virtual Scrolling/Pagination):** หน้าจอแดชบอร์ดที่แสดงรายการเครื่องมือวัดหรือผลสอบเทียบจำนวนมาก (หลักพันรายการขึ้นไป) ต้องใช้การโหลดข้อมูลแบบแบ่งหน้าจากฝั่งเซิร์ฟเวอร์ (Server-side Pagination) หรือใช้ Virtual Scrolling เพื่อประสิทธิภาพการเรนเดอร์หน้าจอที่รวดเร็ว

### 5.5 ด้านการรองรับสองภาษา (Localization & i18n)
* **NFR-5.1 (Dual Language Interface):** ระบบต้องรองรับการสลับภาษาระหว่างภาษาไทยและภาษาอังกฤษได้อย่างสมบูรณ์ โดยห้ามเขียนข้อความแสดงผลฮาร์ดโค้ดในโค้ด UI แต่ให้เรียกใช้ผ่านคลังภาษา i18n
